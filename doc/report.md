### 高等代数 RAG 系统构建方案

背景：现在数学教材的RAG依旧存在很大问题，由于数学教材存在大量符号，跨页公式，箭头等指标；因此OCR表现有限

此外由于数学教材存在大量长证明，长解，因此切分会导致语义缺失，如果在公式处切分，甚至会导致公式无意义，视为乱码

因此我们构造下述RAG系统，针对处理OCR表现和语义处理。

#### 第一阶段：源文档预处理与公式修复

**背景**：MinerU 识别的 Markdown 存在数学符号错乱、LaTeX 语法错误、括号失衡等问题。

**策略**：规则扫描 + LLM 局部重写。

1.  **发现机制（严格规则，宁错杀不放过）**：
    -   `$$...$$` 块交由 PyLaTeX 渲染，报错即提取。
    -   检测 `\left| ... \right|` 或 `\begin{array}` 内部非法出现的 `\hline`。
    -   检测易错模式：`\text{行/列}` 嵌套错误、上下标错位 `[a-z][1-9]`、`\overline`、`\textcircled`。
    -   括号平衡性校验。

2.  **上下文聚合**：
    -   若公式块 `$$...$$` 附近（如紧邻的下一行）出现 `$` 或公式片段，将两者**合并标记**为一个待修复单元 `[TARGET]`，避免切碎公式环境。

3.  **修复执行**：基于 Few-shot 示例（约 21 条规则），调用 LLM 仅对标记的 `[TARGET]` 局部片段进行修正，最大限度保留非公式部分原文。

---

#### 第二阶段：语义感知的文本切分

**技术栈**：LlamaIndex, BGE-M3, Qdrant。

**核心目标**：将教材切分为**语义完整**且**长度可控**的 Chunk，保留层级结构作为上下文。

**切分流线**：

1.  **结构注入**：
    -   识别“第X章 - X.X - X.X.X - 一、”四级标题。
    -   将标题路径写入 Chunk 文本开头作为显式上下文（如：`[第7章 > 7.1 > 7.1.1 一、定义]`）。

2.  **初始切分**：
    -   按**关键词边界**切分：`定义|定理|证明|命题|推论|引理|例|解|解法一`。
    -   逻辑断点处理：
        -   针对“证明”、“解”这类长段落。
        -   若**连续三行**不包含任何数学符号（`$`, `$$`, `\[`, `\begin` 等），判定为纯文字过渡段，于此处断开，标记为 `type: transition`。

3.  **Token 动态调整**：
    -   **过短合并**：Token < 200 的 Chunk，尝试向上合并，若合并后 > 600 则放弃合并。
    -   **过长切分**：Token > 600 的 Chunk（常见于长证明），按逻辑连接词切分（如 `\n由于`, `\n因此`, `\n所以`）。此外，切分起点只有在超过一定字符我们后才能开始，要不然可能会导致过碎，导致语义稀疏

4.  **元数据 ID 分配**：
    -   `block_id`：相同的“证明”或“解”过程内，切分后的多块共享一个 Block ID。
    -   `part_id`：Block 内部按顺序编号（Part 1, Part 2...）。
    -   `id`：最终入库的唯一索引。
    -    [
            {
                "id": "4",
                "metadata": {
                "path": "第7章 一元和 $\\pmb{n}$ 元多项式环 > 7.1 一元多项式环 > 7.1.1 内容精华 > 一、 一元多项式的概念和运算",
                "type": "定义",
                "is_logic_block": false,
                "chapter": "7",
                "is_math_heavy": true,
                "block_id": "group_0",
                "part_id": "1/2",
                "part": "1/2"
                },
                "text": "",
                "token_count": 786
            },
        ]

#### 第三阶段：混合检索与重排序策略

**Baseline_Dense**：纯 Dense 向量检索，固定大小切分。

**Baseline_Hybrid**：混合向量检索，固定大小切分。
chunk_size=512,overlap=50

**Advance 方案**：

1.  **混合检索**：
    -   **Dense Vector**: BGE-M3 稠密向量。
    -   **Sparse Vector**: BGE-M3输出的稀疏向量
    -   **融合**: RRF (Reciprocal Rank Fusion) 取 Top-K。

2.  **重排序扩展策略（对比验证）**：
    -   **模型**: BGE-Reranker-V2-M3 (训练长度 1024 Token)。
    -   **约束**: 单次输入需严格控制在 1024 以内以保证效果。
    -   **方案**:直接将cur_chunk作为rerank的评分指标

3. **生成方案**
rerank得到top-3
模型：qwen3-1.7B

方案A：直接chunk作为上下文
方案B：将chunk中，有block_id的那几个，把具有相同block_id的拿出来，作为上下文

#### 第四阶段：评估指标

1.  **检索质量**：对比 Baseline 与 Hybrid 方案的 **MRR@10, Recall@10**。
2.  **切分质量**：验证 Logic Break 对长证明的可读性提升。
3.  **RAG 生成效果**：接入基座模型，评估问答准确率（Faithfulness）。

4.  **测试集构造**：构造了两个测试集，
    - 测试集A:通常情况的测试集，涵盖单定义，多定义，长短题解/证明
      - 40个单一query，（定理的证明，习题的求解，定理的内容，定义的内容....）
      - 14个对比query(针对每一章节内的定义的对比)
      - 6个多概念对比query
    - 测试集B：针对长证明，测试blockGet的水准
      - 25个长证明，每个answer平均有2000+字符

5. **评价指标**：采用了GLM4.7和deepseek3.2作为LLM-Judgement，从correctness，faithfulness，answer_relevance，context_relevance四个方面来判断，打分为0，1，2，尽可能减少分数之间的模型差异

关于部署的问题

暂时仅支持本地Qdrant数据库，建议使用docker部署
关于Qdrant数据库，建议执行：docker run -d -p 6333:6333 -p 6334:6334 -v "${PWD}/qdrant_storage:/qdrant/storage:z" --name MathRAG qdrant/qdrant

关于config内的参数:
    请仔细阅读configs/config.yaml中的注释
    尤其是api，如果使用.env文件的话，请自行对齐
    如果不打算使用.env文件的话，请直接在config中更换api_key;

关于测试：
    目前仅支持使用llm-judgement，仅支持glm和deepseek（OpenAi格式）的api；虽然deepseek使用的是OpenAi格式，但是字段可能需要自行配置
    注：下述指标基于重构前的核心逻辑版本（v0.1）测得。目前 v1.0 重构版已跑通全流程 Pipeline，由于 LLM-Judgement 运行成本及时间限制，全量测试集的新版本指标正在更新中，但核心性能趋势具有一致性。

关于模型：
    注意：Qwen3-1.7B 性能可能受限，推荐使用 Qwen2.5-7B/14B 或 DeepSeek-V3 以获得最佳生成体验。
